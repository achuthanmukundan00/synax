# v1.0 Local Coding Agent Runtime

Status: Draft

Owner: Unassigned

Target version: v1.0

## Goal

Define the v1.0 bar for Synax as a reliable local coding-agent runtime for developers running local models on consumer GPUs.

## Background

v1.0 should consolidate v0.4 through v0.7 into a coherent local-model-first product. The release should be honest about what Synax does: it should make local models useful and safe for coding work, not claim frontier hosted-agent intelligence or unrestricted autonomy.

## Non-Goals

- No promise to beat Codex on frontier-model intelligence.
- No cloud-agent clone positioning.
- No full intelligent compaction until v1.1.
- No unrestricted autonomy.

## v1.0 Bar

Synax should:

- Work reliably with local Qwen/Unsloth GGUF via Relay/OpenAI-compatible endpoint.
- Survive common malformed tool-call outputs.
- Provide safe bounded repo editing.
- Provide self-docs/spec access.
- Document extension interfaces.
- Include guarded MCP groundwork or a basic bridge, depending on v0.6 outcome.
- Provide patch preview, checkpoint, and rollback safety.
- Provide verification profiles.
- Provide a readable CLI/TUI enough for daily use.
- Provide provider/model compatibility diagnostics.
- Publish docs that clearly explain local-model-first positioning.
- Provide a compatibility matrix or diagnostic report format.
- Ship with a release checklist.

- v0.6 Extension Kernel + MCP Bridge Groundwork completed or explicitly descoped with alternatives.

## Compatibility Matrix / Diagnostic Report

Define a public format that can record:

- provider/gateway name and version when known
- model name and quantization when known
- endpoint compatibility
- native tool-call behavior
- text-shaped tool-call behavior
- reasoning leakage behavior
- max context used during smoke test
- pass/fail diagnostics
- known caveats

The matrix may begin as a docs table or CLI-generated report, but it must distinguish verified behavior from assumptions.

## Release Checklist

- [ ] `package.json` version intentionally updated for the release.
- [ ] README matches implemented behavior.
- [ ] VitePress docs match implemented behavior.
- [ ] Specs and progress log reflect completed scope.
- [ ] Compatibility matrix or diagnostic report exists.
- [ ] `npm run typecheck` passes.
- [ ] `npm test` passes.
- [ ] `npm run build` passes.
- [ ] `npm run docs:build` passes.
- [ ] Smoke test with a local Relay-compatible endpoint is documented, or skipped with a concrete reason.
- [ ] Known limitations are documented.

## Tests

- Full local test suite.
- CLI smoke tests for core commands.
- Parser torture suite.
- Config and provider diagnostics tests.
- Patch safety tests.
- Docs/spec provider tests.
- Extension interface compile tests.
- Optional local Relay smoke test when available.

## Docs Impact

- README should explain install, configure, Relay setup, local-model expectations, commands, safety, verification, and limitations.
- VitePress docs should cover getting started, Relay, configuration, commands, agent loop, safety/context, development, compatibility, and release notes if present.
- Public docs must avoid claims that require cloud APIs or frontier-model behavior.

## Acceptance Criteria

- [ ] Core workflows work without OpenAI-hosted APIs.
- [ ] Local-model malformed output cases are tested and documented.
- [ ] Edits are previewable, bounded, and recoverable.
- [ ] Verification profiles are documented and usable.
- [ ] Extension interfaces are documented.
- [ ] MCP status is clear: groundwork only or basic bridge.
- [ ] Compatibility report format exists.
- [ ] Release checklist passes or documents explicit exceptions.

## Verification Commands

```sh
npm run typecheck
npm test
npm run build
npm run docs:build
```

## Rollback Plan

If v1.0 criteria are not met, do not ship the v1.0 label. Keep the package on a pre-v1 version, document the missing criteria in `specs/PROGRESS.md`, and release only a smaller version if the implemented scope is coherent.
