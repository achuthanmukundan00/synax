# v0.4 Tool-Call Survival

Status: Draft

Owner: Unassigned

Target version: v0.4

## Goal

Make Synax robust against common local-model tool-call failures while preserving safe failure behavior.

## Background

Synax already sends OpenAI-compatible tools and currently accepts native `message.tool_calls` plus simple text fallback formats such as `<tool_call>{...}</tool_call>`. v0.4 should harden this into a tested parser/normalizer path aimed at Qwen/Unsloth GGUF, llama.cpp, and Relay behavior.

Local models may emit malformed JSON, stringified arguments, fenced JSON, leaked reasoning tags, mixed final answers, or multiple competing tool-call shapes. Synax should recover only when intent is unambiguous.

## Non-Goals

- No full MCP.
- No full TUI.
- No intelligent compaction.
- No provider rewrite.

## Implementation Plan

1. Inspect existing parser behavior and tests.
2. Convert or wrap the current parser into a dedicated tool-call parser/normalizer module with explicit result types.
3. Support native OpenAI `message.tool_calls`.
4. Support `<tool_call>{...}</tool_call>` blocks.
5. Support fenced JSON only when the block is unambiguously a tool call.
6. Accept `arguments` as either an object or stringified JSON object.
7. Add sanitizer handling for `<think>...</think>` and similar reasoning leakage before parsing or rendering final answers.
8. Reject unknown tools before execution.
9. Fail safely when multiple incompatible tool calls or mixed final/tool output make intent ambiguous.
10. Add parser torture fixtures for valid, malformed, ambiguous, unknown, reasoning-leak, Qwen-style, and stringified-argument cases.
11. Add a diagnostic or smoke command if it fits the existing CLI shape, likely under `doctor` or a narrow test-provider path.
12. Update docs with supported formats, rejected formats, diagnostics, and local-model caveats.
13. Update `specs/PROGRESS.md` and `specs/LEARNINGS.md`.

## Parser Requirements

- Native OpenAI calls: parse `message.tool_calls[*].function.name` and `.function.arguments`.
- XML-ish calls: parse `<tool_call>{...}</tool_call>`.
- Fenced JSON: parse only when the block clearly contains a single tool call or OpenAI-style tool call list.
- Arguments: accept plain JSON objects and stringified JSON objects.
- Reasoning tags: remove or ignore `<think>...</think>`, `<thinking>...</thinking>`, and similar known local-model reasoning wrappers.
- Unknown tools: reject with a clear parser/tool-policy result; do not execute.
- Ambiguous tool calls: fail closed with a useful diagnostic.
- Mixed final answer plus tool call: execute only when the tool call is unambiguous and policy allows it; otherwise fail closed.

## Tests

- Native OpenAI `message.tool_calls` with object arguments.
- Native OpenAI `message.tool_calls` with stringified arguments.
- Qwen-style `<tool_call>` block.
- Fenced JSON single tool call.
- Fenced JSON that is normal prose data and must not execute.
- Leaked `<think>` block before a valid tool call.
- Leaked reasoning in final answer rendering.
- Unknown tool name rejection.
- Ambiguous multiple formats rejection.
- Malformed JSON rejection with clear error.
- Parser does not require a live model server.

## Diagnostic / Smoke Command

If it fits the existing CLI shape, extend `doctor` or `/test-provider` to report whether a configured provider can produce or accept the expected local tool-call pattern. This should be diagnostic-only and should not require editing files.

## Acceptance Criteria

- [ ] Parser/normalizer has explicit typed success and failure results.
- [ ] Tests cover native, Qwen-style, fenced, stringified, malformed, unknown, reasoning-leak, and ambiguous cases.
- [ ] Unknown or ambiguous tool calls are not executed.
- [ ] Reasoning/thinking leakage is sanitized from final user-visible output where applicable.
- [ ] `npm run typecheck` passes.
- [ ] `npm test` passes.
- [ ] `npm run build` passes.
- [ ] Docs describe implemented support without overclaiming.
- [ ] `specs/PROGRESS.md` is updated.
- [ ] `specs/LEARNINGS.md` is updated with any real compatibility findings.

## Rollback Plan

Revert parser/normalizer changes and docs updates together. Restore the previous `src/llm/tool-calls.ts` behavior and ensure existing tests still pass. Do not leave docs claiming support that has been rolled back.
