# v0.7 Daily-Driver Safety

Status: Draft

Owner: Unassigned

Target version: v0.7

## Goal

Make Synax safe enough for routine repo work by adding preview, checkpoint, rollback, logs, and bounded verification behavior.

## Background

Synax currently emphasizes bounded tools and safe defaults. v0.7 should improve user trust around edits: the user should be able to see what will change, accept or reject it, recover from the last edit/task, and understand verification status.

## Non-Goals

- No full fancy TUI if simple CLI flows are enough.
- No autonomous unbounded editing.
- No destructive git operations without explicit user command.

## Required Features

- Patch preview before application.
- Accept/reject flow.
- Atomic patch application.
- `/diff`.
- `/undo-last-edit` or equivalent.
- Git checkpoint before run/task.
- Dirty tree handling.
- Optional worktree mode.
- Run logs under `.synax/runs/` or equivalent.
- Verification profiles and bounded repair loop.
- Readable status summaries.

## Implementation Plan

1. Inspect current patch application, chat commands, run command, and verification behavior.
2. Define a patch preview data shape shared by chat/run renderers.
3. Add accept/reject flow with non-interactive behavior defined for `run`.
4. Ensure patch application is atomic or fails without partial writes.
5. Add `/diff` for current run/task state.
6. Add `/undo-last-edit` or equivalent rollback of the most recent Synax edit.
7. Add git checkpoint before run/task without destructive operations.
8. Define dirty tree behavior and require explicit user consent for risky flows.
9. Add optional worktree mode if it fits current config and command architecture.
10. Write run logs under `.synax/runs/` or equivalent, excluding sensitive data.
11. Add verification profiles and a bounded repair loop.
12. Update docs and tests.

## Safety Requirements

- Dirty working trees must be detected and summarized.
- Synax must not run `git reset --hard`, `git clean`, or destructive git operations unless explicitly commanded by the user.
- Rollback must be scoped to Synax edits, not arbitrary user changes.
- Run logs must not include API keys or full prompts by default.
- Repair loops must have explicit limits.

## Tests

- Patch preview renders intended changes.
- Accept applies the exact previewed patch.
- Reject leaves files unchanged.
- Failed patch does not leave partial file content.
- `/diff` shows bounded current changes.
- `/undo-last-edit` restores the last Synax edit without touching unrelated changes.
- Dirty tree handling is tested.
- Verification profile and repair-loop limits are tested.

## Docs Impact

- Update command docs for preview, accept/reject, `/diff`, and undo.
- Update safety docs for checkpoint, dirty tree, worktree mode, logs, and rollback.
- Update development docs for verification profiles.
