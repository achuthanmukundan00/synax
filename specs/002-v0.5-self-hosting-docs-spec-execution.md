# v0.5 Self-Hosting Docs + Spec Execution

Status: Draft

Owner: Unassigned

Target version: v0.5

## Goal

Make Synax better at modifying itself and its docs by giving it bounded local access to project docs/specs and a practical spec-execution workflow.

## Background

Synax already has README and VitePress docs, plus this `specs/` scaffold. `synax run --plan plan.md` is currently documented as a placeholder. v0.5 should turn self-hosting into a concrete, bounded workflow without adding embeddings, MCP, or autonomous project management.

## Non-Goals

- No full RAG system.
- No embeddings requirement.
- No autonomous long-running project management.
- No MCP dependency.

## Implementation Plan

1. Inspect current `run`, `inspect`, and tool registry behavior.
2. Add a local docs provider covering `README.md`, `docs/**`, `AGENTS.md`, `CHANGELOG.md` if present, `specs/**`, and config examples.
3. Add simple docs search/read capability using file names, headings, and text search; do not require embeddings.
4. Design and implement plan/spec execution support such as `synax run --plan specs/...` if it fits current CLI architecture.
5. Add a docs impact checker for source/config/CLI/tool behavior changes.
6. Define the self-hosting workflow: inspect spec, inspect code/docs, patch, verify, update progress/learnings.
7. Update README and VitePress docs only for behavior actually implemented.
8. Add tests for docs discovery, bounded reads, plan/spec handling, and docs impact checks.

## Self-Hosting Workflow

The planned workflow should be:

1. Inspect the target spec.
2. Inspect relevant code and docs.
3. Produce a compact plan.
4. Patch the smallest implementation surface.
5. Run targeted verification.
6. Update public docs when behavior changes.
7. Update `specs/PROGRESS.md`.
8. Update `specs/LEARNINGS.md` when new local-model or workflow facts were learned.

## Tests

- Docs provider finds README, VitePress docs, AGENTS.md, specs, and config examples.
- Docs provider excludes generated docs output such as `docs/.vitepress/dist/`.
- Docs reads are bounded.
- `synax run --plan specs/...` either executes a supported path or fails with a clear not-yet-supported message until implemented.
- Docs impact checker flags source/config/CLI/tool changes that likely require docs updates.
- Tests do not require network or live model server.

## Docs Impact

- Update `README.md` if `run --plan` becomes implemented.
- Update `docs/guide/commands.md` for command behavior.
- Update `docs/guide/development.md` for the self-hosting workflow.
- Update `docs/guide/safety-context.md` if docs context affects budgeting or file policy.

## Acceptance Criteria

- [ ] Synax can discover and read local docs/spec files through bounded mechanisms.
- [ ] Generated docs output is excluded.
- [ ] Plan/spec execution behavior is implemented or explicitly scoped to a clear placeholder with tests.
- [ ] Docs impact checks exist for public behavior changes.
- [ ] Self-hosting workflow is documented.
- [ ] `npm run typecheck` passes.
- [ ] `npm test` passes.
- [ ] `npm run build` passes.
- [ ] `npm run docs:build` passes.
- [ ] `specs/PROGRESS.md` is updated.
- [ ] `specs/LEARNINGS.md` is updated when applicable.

## Verification Commands

```sh
npm run typecheck
npm test
npm run build
npm run docs:build
```

## Rollback Plan

Revert docs provider, plan/spec execution, docs impact checker, tests, and docs together. If `run --plan` is rolled back, restore docs that describe it as a placeholder.
