# v0.6 Extension Kernel + MCP Bridge Groundwork

Status: Draft

Owner: Unassigned

Target version: v0.6

## Goal

Define stable TypeScript extension interfaces and safe MCP bridge groundwork without turning Synax into a plugin marketplace.

## Background

Synax should remain small and inspectable, but future compatibility work needs stable internal seams. v0.6 should name those seams, add minimal wiring, and establish MCP safety boundaries before exposing broad integration behavior.

## Non-Goals

- No arbitrary plugin marketplace.
- No dynamic remote code execution.
- No broad MCP server implementation.
- No unrestricted shell via MCP.

## Files Likely Involved

- `src/llm/`
- `src/tools/`
- `src/agent/`
- `src/config/`
- `src/commands/`
- new extension interface modules if needed
- `src/__tests__/`
- `docs/guide/development.md`
- `docs/guide/agent-loop.md`
- `docs/guide/safety-context.md`
- `specs/PROGRESS.md`
- `specs/LEARNINGS.md`

## Required Interfaces

Define stable TypeScript interfaces for:

- `ToolCallParser`
- `ToolCallRepairer`
- `ReasoningSanitizer`
- `ProviderAdapter`
- `ContextProvider`
- `Verifier`
- `DocsProvider`
- `Renderer`
- `McpBridge`

Interfaces should be minimal, documented, and driven by existing implementation needs. Do not add abstraction for its own sake.

## Extension Registry / Wiring

Add either a small extension registry or an internal wiring pattern. It should:

- wire built-in implementations explicitly
- avoid global mutable plugin state
- be testable without network access
- preserve existing CLI behavior unless a command intentionally changes
- make future additions visible in one or two obvious places

## MCP Bridge Groundwork

Define concepts for:

- Native Synax tool to MCP export.
- Guarded MCP tool import.
- Docs/specs as MCP resources.
- Prompt packs as MCP prompts.

This may start as interfaces and documentation before runtime bridge behavior exists.

## Safety Boundaries

- MCP must not bypass Synax tool policy.
- MCP must not bypass approval/checkpoint policy.
- MCP must not bypass verification policy.
- MCP must not bypass context/budget policy.
- MCP must not expose unrestricted shell.
- MCP-imported tools must have explicit schemas, names, and policy checks.

## Tests

- Interface compile tests or narrow unit tests for built-in wiring.
- Registry tests that prove built-ins are available.
- Policy tests showing imported tools cannot bypass existing tool restrictions.
- Docs/spec resource tests if resources are implemented.
- No live MCP server should be required unless explicitly added as an optional smoke check.

## Docs Impact

- Document extension interfaces for maintainers.
- Document MCP as groundwork if runtime bridge behavior is not yet implemented.
- Update safety docs for MCP boundaries.
- Avoid advertising marketplace or arbitrary plugin support.

## Acceptance Criteria

- [ ] Required interfaces exist and are documented.
- [ ] Built-in behavior is wired through the minimal registry or internal pattern.
- [ ] Tests cover wiring and safety boundaries.
- [ ] MCP groundwork is clearly marked as groundwork unless runtime behavior exists.
- [ ] `npm run typecheck` passes.
- [ ] `npm test` passes.
- [ ] `npm run build` passes.
- [ ] `npm run docs:build` passes.
- [ ] `specs/PROGRESS.md` is updated.
- [ ] `specs/LEARNINGS.md` is updated when applicable.

## Verification Commands

```sh
npm run typecheck
npm test
npm run build
npm run docs:build
```

## Rollback Plan

Revert interface, wiring, MCP groundwork, tests, and docs together. Keep v0.4/v0.5 behavior intact.
